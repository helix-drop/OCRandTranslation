## 当前限制

### 1. 并发翻译尚未开始

当前流式页翻译是顺序逐段执行，不做并发。

原因：

- 先保证停止可控
- 先保证 UI 与物理页导航一致
- 避免并发后多页状态混乱

### 2. 草稿刷新仍是前端内存态

当前流式草稿显示依赖阅读页当前会话的 SSE。

现状：

- 刷新页面后，终态和页级进度能恢复
- 但“当前段落草稿文本”不会从磁盘恢复

### 3. 失败页状态尚未进入导航体系

当前页码选择器已支持：

- 已译
- 待译
- 翻译中
- 已中断

但尚未将“失败页”单独标记为导航状态。

### 4. 增量存储仍未开始

当前已消除阅读路由上的整文件重写，但翻译结果仍然整体写入 `entries.json`。

这意味着：

- 阅读性能已明显改善
- 但大文档长时间翻译时，写盘仍不是最优

## 待评估方案

### 卷轴滚动式 PDF 阅读

#### 问题

当前逐页点击的阅读方式存在局限：
- 频繁点击打断阅读流
- 难以快速浏览全文结构
- 跨页内容（表格、图表分割）查看不便

#### 目标

用卷轴滚动替代分页点击，同时保留"页"的概念用于翻译对照。

#### 方案设计

**布局：双栏卷轴 + 页同步**

```
┌─────────────────┬─────────────────┐
│   PDF 卷轴视图   │   翻译对照区     │
│  (连续滚动)      │  (当前页翻译)   │
│                 │                 │
│ ┌───────────┐   │ ┌───────────┐  │
│ │  第1页     │   │ │  原文      │  │
│ └───────────┘   │ │  译文      │  │
│ ┌───────────┐   │ └───────────┘  │
│ │  第2页     │   │ [下页翻译 ▶]   │
│ └───────────┘   │                 │
│ ...             │                 │
└─────────────────┴─────────────────┘
     ↑ 滚动位置同步 → 显示当前可见页翻译
```

**交互细节**

| 功能 | 行为 |
|------|------|
| PDF 区 | 卷轴滚动，页与页之间有明显分隔线（显示页码标记） |
| 翻译区 | 显示 PDF 视口中心所在页的翻译 |
| 页码导航 | 保留页码输入框，跳转时 PDF 滚动到对应位置 |
| 进度条 | 右侧显示翻译进度标记，可点击跳转到已译页 |
| 翻译触发 | 滚动到未译页时，可点击"翻译当前页"按钮 |

**技术实现**

选项 A：PDF.js viewer
- 使用 PDF.js 内置的滚动查看器
- 优点：成熟稳定，支持文字选择
- 缺点：与当前 PyMuPDF 渲染体系冲突

选项 B：自研虚拟滚动
- PyMuPDF 渲染 + Intersection Observer 检测当前可见页
- 优点：与现有架构兼容，可控性强
- 缺点：需自行实现页码标记、滚动性能优化

**性能考虑**

- 大 PDF 不能一次性渲染所有页
- 采用懒加载/虚拟滚动：只渲染视口上下 N 页
- 页码标记需预计算，用于滚动条定位

**待解决问题**

1. 流式翻译时，草稿区如何与卷轴位置同步？
2. 后台翻译进度条与卷轴阅读如何共存？
3. 双栏布局在移动端如何适配？

#### 优先级

低优先级 - 当前分页模式已满足基本需求，卷轴模式作为体验优化待后续评估。
